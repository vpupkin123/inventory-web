1. Обзор системы

Inventory Web — это лёгкое веб-приложение для учёта компьютерного оборудования, разработанное специально для работы на NAS Synology без использования внешних зависимостей.

Ключевые особенности:

  • Минимализм: без фреймворков, без Composer, без тяжёлых зависимостей
  • Автономность: всё необходимое встроено — PHP + SQLite
  • Многоязычность: русский и английский интерфейсы
  • Безопасность: хеширование паролей, защита от SQL-инъекций, обязательная смена пароля при первом входе
  • Совместимость: корректная работа с файлами из Windows (обработка BOM, кириллицы)
  • Автоматизация: автогенерация логинов по ГОСТ Р 7.0.34-2014

2. Стек технологий

Серверная часть:

  • PHP 8.0+ — основной язык
  • SQLite 3 — база данных (один файл inventory.db)
  • HTML5 — разметка
  • JavaScript — интерактивность (валидация, AJAX)
  • CSS3 — стили (встроенные, без фреймворков)

Архитектура:

  • MVC-подобная структура — разделение на контроллеры, представления и модели
  • Единая точка входа — public/index.php + .htaccess для ЧПУ
  • Собственный роутер — класс App обрабатывает маршруты
  • Автозагрузчик классов — Autoloader для сервисов и контроллеров
  • Система интернационализации — класс Lang с файлами переводов

Внешние зависимости:

Отсутствуют. Никаких библиотек, Composer, CDN — всё работает автономно.

3. Системные требования

Минимальные:

Компонент Требование
Операционная система Linux (проверено на Synology DSM 7+)
Веб-сервер Apache 2.4+ с mod_rewrite или Nginx
PHP 8.0 или выше (с расширениями: PDO, SQLite, JSON, mbstring)
SQLite 3.x (встроен в PHP)
Диск ~10 МБ для приложения + место для БД и загрузок
Память ~50 МБ RAM
💡 Проверено на: Synology DS920+ с Web Station и PHP 8.0. Работает также на любых других Linux-серверах с аналогичным стеком.

4. Установка на Synology

Шаг 1: Подготовка Web Station

  1. Установите Web Station из Package Center
  2. Установите Apache 2.4 и PHP 8.0 (или выше)
  3. Включите модуль mod_rewrite в настройках Apache

Шаг 2: Клонирование репозитория

По SSH подключитесь к Synology и выполните:

cd /volume1/web
git clone https://github.com/vpupkin123/inventory-web.git
cd inventory-web

Шаг 3: Настройка виртуального хоста

  1. Откройте Web Station → Портал веб-сайтов
  2. Создайте новый виртуальный хост:
    • Имя сервера: inventory-web (или ваш домен)
    • Корневая папка: /volume1/web/inventory-web/public
    • PHP: PHP 8.0
    • HTTP-сервер: Apache 2.4
  3. В настройках DNS или /etc/hosts пропишите имя сервера

Шаг 4: Настройка прав доступа

В папке admin-scripts/ находится универсальный скрипт fix-permissions.sh для настройки прав.

Сделайте его исполняемым и запустите:

chmod +x /volume1/web/inventory-web/admin-scripts/fix-permissions.sh
cd /volume1/web/inventory-web
./admin-scripts/fix-permissions.sh
💡 Универсальность: Скрипт автоматически определяет текущего пользователя и группу. При необходимости можно передать параметры: 
./admin-scripts/fix-permissions.sh /path/to/project owner group webuser

Шаг 5: Первая инициализация

При первом обращении к сайту автоматически:

  • Создаётся база данных data/inventory.db
  • Создаётся виртуальный пользователь warehouse (склад)
  • Создаётся администратор: логин admin, пароль password
🚨 ОБЯЗАТЕЛЬНО: Сразу после первого входа смените пароль администратора!

6. База данных

SQLite-файл data/inventory.db содержит 4 таблицы:

Таблица users

ПолеТипОписание
idINTEGERПервичный ключ
loginTEXT UNIQUEУникальный логин
password_hashTEXTХеш пароля (bcrypt)
last_name, first_name, middle_nameTEXTФИО
roleTEXTadmin / editor / viewer / none
must_change_pwdINTEGER1 = нужно сменить пароль
created_atTEXTДата создания

Таблица computers

ПолеТипОписание
idINTEGERПервичный ключ
serial_numberTEXTСерийный номер (или NULL)
composite_keyTEXT UNIQUEХеш для ПК без серийника
manufacturer, modelTEXTПроизводитель и модель
motherboard, cpu_name, ram_total_gbTEXT/REALОсновные характеристики
storage_info, ram_detailsTEXTДетали накопителей и RAM
os_caption, os_buildTEXTОперационная система
ip_address, computer_nameTEXTСетевые данные
reported_byTEXTФИО из отчёта
current_user_idINTEGER FKСсылка на владельца
commentTEXTКомментарий администратора
is_processedINTEGER0 = в очереди, 1 = обработан

Таблица transfers

Журнал всех перемещений оборудования. Хранит историю: от кого, кому, кем и когда передан каждый ПК.

Таблица processed_uploads

Реестр загруженных JSON-файлов с хешами для защиты от дубликатов.

✅ Миграции: При обновлении версии класс Database автоматически добавляет недостающие поля через ALTER TABLE.

7. Настройка прав доступа

Принцип:

  • Пользователь Synology (zaikoz) — владелец проекта, может делать git pull
  • Пользователь http — веб-сервер, должен иметь запись в data/
  • Группа administrators — для совместного доступа

Таблица прав:

ПутьВладелецПраваПричина
Весь проект zaikoz:administrators 755 (папки), 644 (файлы) Git pull, редактирование
data/ http:http 775 Запись БД, логов, загрузок
data/inventory.db http:http 664 Запись в БД
data/uploads/ http:http 775 Загрузка JSON-файлов
⚠️ После каждого git pull права могут сбиваться. Всегда запускайте ./admin-scripts/fix-permissions.sh после обновления.

8. Роли и права пользователей

РольДоступ
admin Полный доступ: всё, включая управление пользователями
editor Загрузка отчётов, обработка, передача ПК, просмотр всего
viewer Только просмотр данных (без изменений)
none Нет доступа (заблокирован) или виртуальный пользователь (склад)
💡 Виртуальный пользователь warehouse: имеет роль none, используется как "владелец" для компьютеров на складе. Не может входить в систему.

9. Конфигурация

Основные настройки находятся в config/config.php:

<?php
define('ROOT_PATH', dirname(__DIR__));
define('DATA_PATH', ROOT_PATH . '/data');
define('DB_FILE', DATA_PATH . '/inventory.db');
define('WAREHOUSE_LOGIN', 'warehouse');
define('APP_VERSION', '1.0.0');

Ключевые константы:

  • ROOT_PATH — корень проекта
  • DATA_PATH — папка для данных
  • DB_FILE — путь к файлу БД
  • WAREHOUSE_LOGIN — логин виртуального склада

10. Логирование

Приложение ведёт лог ошибок в файле data/app_error.log.

Что логируется:

  • PHP-ошибки и исключения
  • Ошибки при загрузке отчётов
  • Ошибки при передаче компьютеров

Просмотр логов:

tail -f /volume1/web/inventory-web/data/app_error.log
✅ Совет: Настройте ротацию логов, чтобы файл не разрастался. Например, через cron: 
0 0 * * 0 mv /volume1/web/inventory-web/data/app_error.log /volume1/web/inventory-web/data/app_error.log.$(date +\%Y\%m\%d)

11. Резервное копирование

Для полного бэкапа достаточно скопировать папку data/:

# Создание бэкапа
tar -czf inventory-backup-$(date +%Y%m%d).tar.gz \
    /volume1/web/inventory-web/data/

# Восстановление
tar -xzf inventory-backup-YYYYMMDD.tar.gz -C /

Что входит в бэкап:

  • inventory.db — вся база данных
  • uploads/ — все загруженные JSON-отчёты
  • app_error.log — логи (опционально)
⚠️ Важно: Сам код приложения хранится в Git и может быть восстановлен из репозитория. Бэкапить нужно только data/.

12. Обновление системы

Стандартная процедура:

cd /volume1/web/inventory-web
git stash              # Сохранить локальные изменения (если есть)
git pull               # Получить обновления из репозитория
./admin-scripts/fix-permissions.sh   # Восстановить права

Если были локальные правки:

git stash pop          # Вернуть локальные изменения
💡 Миграции БД: При обновлении новые поля в таблицах создаются автоматически через метод Database::migrate(). Никаких ручных действий не требуется.

13. Решение проблем

Ошибка 500 (Internal Server Error)

Причина: PHP-ошибка. Проверьте лог:

cat /volume1/web/inventory-web/data/app_error.log

Частые причины:

  • Отсутствует файл (например, Lang.php) — проверьте git status
  • Дубликат метода в классе — проверьте синтаксис через php -l файл.php
  • Нет прав на запись в data/ — запустите ./admin-scripts/fix-permissions.sh

Ошибка 404 на новых страницах

Причина: не добавлен маршрут в config/routes.php или отсутствует контроллер.

Решение: проверьте наличие файла контроллера и строки в роутах.

Не загружаются JSON-файлы

Причины:

  • Нет прав на запись в data/uploads/ — исправьте права
  • Файл содержит BOM — парсер автоматически удаляет BOM, но если ошибка остаётся, проверьте app_error.log
  • Файл уже загружен — это нормальная защита от дубликатов

Не работает mod_rewrite

Симптомы: все URL кроме / возвращают 404.

Решение:

  1. Убедитесь, что mod_rewrite включён в Apache
  2. Проверьте наличие .htaccess в public/
  3. В настройках виртуального хоста разрешите AllowOverride All

Сброс базы данных (для тестирования)

Используйте скрипт admin-scripts/reset-inventory.sh:

cd /volume1/web/inventory-web
./admin-scripts/reset-inventory.sh
Осторожно: это удалит все компьютеры, отчёты и передачи. Пользователи останутся.
