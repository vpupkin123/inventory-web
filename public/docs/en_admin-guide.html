1. System Overview

Inventory Web is a lightweight web application for tracking computer equipment, specifically designed to run on Synology NAS without external dependencies.

Key Features:

  • Minimalism: no frameworks, no Composer, no heavy dependencies
  • Autonomy: everything needed is built-in — PHP + SQLite
  • Multi-language: Russian and English interfaces
  • Security: password hashing, SQL injection protection, mandatory password change on first login
  • Compatibility: correct handling of files from Windows (BOM processing, Cyrillic support)
  • Automation: auto-generation of logins according to GOST R 7.0.34-2014

2. Technology Stack

Server-side:

  • PHP 8.0+ — main language
  • SQLite 3 — database (single inventory.db file)
  • HTML5 — markup
  • JavaScript — interactivity (validation, AJAX)
  • CSS3 — styles (embedded, no frameworks)

Architecture:

  • MVC-like structure — separation into controllers, views, and models
  • Single entry point — public/index.php + .htaccess for clean URLs
  • Custom router — App class handles routes
  • Class autoloader — Autoloader for services and controllers
  • Internationalization system — Lang class with translation files

External Dependencies:

None. No libraries, no Composer, no CDN — everything works autonomously.

3. System Requirements

Minimum:

Component Requirement
Operating System Linux (tested on Synology DSM 7+)
Web Server Apache 2.4+ with mod_rewrite or Nginx
PHP 8.0 or higher (with extensions: PDO, SQLite, JSON, mbstring)
SQLite 3.x (built into PHP)
Disk ~10 MB for the application + space for DB and uploads
Memory ~50 MB RAM
💡 Tested on: Synology DS920+ with Web Station and PHP 8.0. Also works on any other Linux servers with a similar stack.

4. Installation on Synology

Step 1: Prepare Web Station

  1. Install Web Station from Package Center
  2. Install Apache 2.4 and PHP 8.0 (or higher)
  3. Enable mod_rewrite module in Apache settings

Step 2: Clone Repository

Connect to Synology via SSH and execute:

cd /volume1/web
git clone https://github.com/vpupkin123/inventory-web.git
cd inventory-web

Step 3: Configure Virtual Host

  1. Open Web Station → Web Portal
  2. Create a new virtual host:
    • Server name: inventory-web (or your domain)
    • Root folder: /volume1/web/inventory-web/public
    • PHP: PHP 8.0
    • HTTP Server: Apache 2.4
  3. In DNS settings or /etc/hosts, configure the server name

Step 4: Configure Access Permissions

The admin-scripts/ folder contains a universal script fix-permissions.sh for setting permissions.

Make it executable and run:

chmod +x /volume1/web/inventory-web/admin-scripts/fix-permissions.sh
cd /volume1/web/inventory-web
./admin-scripts/fix-permissions.sh
💡 Universal: The script automatically detects the current user and group. If needed, you can pass parameters: 
./admin-scripts/fix-permissions.sh /path/to/project owner group webuser

Step 5: First Initialization

On first access to the site, automatically:

  • Database data/inventory.db is created
  • Virtual user warehouse is created
  • Administrator is created: login admin, password password
🚨 MANDATORY: Immediately after first login, change the administrator password!

6. Database

SQLite file data/inventory.db contains 4 tables:

Table users

FieldTypeDescription
idINTEGERPrimary key
loginTEXT UNIQUEUnique login
password_hashTEXTPassword hash (bcrypt)
last_name, first_name, middle_nameTEXTFull name
roleTEXTadmin / editor / viewer / none
must_change_pwdINTEGER1 = must change password
created_atTEXTCreation date

Table computers

FieldTypeDescription
idINTEGERPrimary key
serial_numberTEXTSerial number (or NULL)
composite_keyTEXT UNIQUEHash for PCs without serial
manufacturer, modelTEXTManufacturer and model
motherboard, cpu_name, ram_total_gbTEXT/REALMain specifications
storage_info, ram_detailsTEXTStorage and RAM details
os_caption, os_buildTEXTOperating system
ip_address, computer_nameTEXTNetwork data
reported_byTEXTFull name from report
current_user_idINTEGER FKReference to owner
commentTEXTAdministrator comment
is_processedINTEGER0 = in queue, 1 = processed

Table transfers

Log of all equipment movements. Stores history: from whom, to whom, by whom, and when each PC was transferred.

Table processed_uploads

Registry of uploaded JSON files with hashes for duplicate protection.

✅ Migrations: When updating the version, the Database class automatically adds missing fields via ALTER TABLE.

7. Access Permissions Setup

Principle:

  • Synology user (admin) — project owner, can do git pull
  • User http — web server, must have write access to data/
  • Group administrators — for shared access

Permissions Table:

PathOwnerPermissionsReason
Entire project zaikoz:administrators 755 (folders), 644 (files) Git pull, editing
data/ http:http 775 Write DB, logs, uploads
data/inventory.db http:http 664 Write to DB
data/uploads/ http:http 775 Upload JSON files
⚠️ After each git pull permissions may reset. Always run ./admin-scripts/fix-permissions.sh after updating.

8. User Roles and Permissions

RoleAccess
admin Full access: everything, including user management
editor Upload reports, processing, computer transfers, view all
viewer View-only access (no changes)
none No access (blocked) or virtual user (warehouse)
💡 Virtual user warehouse: has role none, used as "owner" for computers on the warehouse. Cannot log into the system.

9. Configuration

Main settings are in config/config.php:

<?php
define('ROOT_PATH', dirname(__DIR__));
define('DATA_PATH', ROOT_PATH . '/data');
define('DB_FILE', DATA_PATH . '/inventory.db');
define('WAREHOUSE_LOGIN', 'warehouse');
define('APP_VERSION', '1.0.0');

Key Constants:

  • ROOT_PATH — project root
  • DATA_PATH — data folder
  • DB_FILE — path to DB file
  • WAREHOUSE_LOGIN — virtual warehouse login

10. Logging

The application maintains an error log in data/app_error.log.

What is logged:

  • PHP errors and exceptions
  • Errors during report uploads
  • Errors during computer transfers

View logs:

tail -f /volume1/web/inventory-web/data/app_error.log
✅ Tip: Set up log rotation to prevent the file from growing. For example, via cron: 
0 0 * * 0 mv /volume1/web/inventory-web/data/app_error.log /volume1/web/inventory-web/data/app_error.log.$(date +\%Y\%m\%d)

11. Backup

For a complete backup, it's sufficient to copy the data/ folder:

# Create backup
tar -czf inventory-backup-$(date +%Y%m%d).tar.gz \
    /volume1/web/inventory-web/data/

# Restore
tar -xzf inventory-backup-YYYYMMDD.tar.gz -C /

What's included in the backup:

  • inventory.db — entire database
  • uploads/ — all uploaded JSON reports
  • app_error.log — logs (optional)
️ Important: The application code itself is stored in Git and can be restored from the repository. Only data/ needs to be backed up.

12. System Update

Standard procedure:

cd /volume1/web/inventory-web
git stash              # Save local changes (if any)
git pull               # Get updates from repository
./admin-scripts/fix-permissions.sh   # Restore permissions

If there were local changes:

git stash pop          # Restore local changes
💡 DB Migrations: When updating, new fields in tables are created automatically via the Database::migrate() method. No manual actions required.

13. Troubleshooting

Error 500 (Internal Server Error)

Cause: PHP error. Check the log:

cat /volume1/web/inventory-web/data/app_error.log

Common causes:

  • Missing file (e.g., Lang.php) — check git status
  • Duplicate method in class — check syntax via php -l file.php
  • No write permissions to data/ — run ./admin-scripts/fix-permissions.sh

Error 404 on new pages

Cause: route not added in config/routes.php or controller missing.

Solution: check for controller file existence and route line.

JSON files not uploading

Causes:

  • No write permissions to data/uploads/ — fix permissions
  • File contains BOM — parser automatically removes BOM, but if error persists, check app_error.log
  • File already uploaded — this is normal duplicate protection

mod_rewrite not working

Symptoms: all URLs except / return 404.

Solution:

  1. Ensure mod_rewrite is enabled in Apache
  2. Check for .htaccess in public/
  3. In virtual host settings, allow AllowOverride All

Reset database (for testing)

Use script admin-scripts/reset-inventory.sh:

cd /volume1/web/inventory-web
./admin-scripts/reset-inventory.sh
🚨 Caution: this will delete all computers, reports, and transfers. Users will remain.
